### 2.2 功能详细设计

#### ① 外卖/访客按键提醒

```
用户场景: 外卖员按门外按键 → 室内立刻知道
         屋主不在 → 记录来访时间

工作流:
  ┌──────────┐     ┌──────────┐     ┌──────────┐
  │ GPIO 检测  │────▶ Agent 判断│────▶ 通知执行   │
  │ 按键下降沿 │     │ · 是否在家 │     │ · TTS 播报  │
  │ 防抖处理  │     │ · 时间段   │     │ · 屏显通知  │
  └──────────┘     │ · 是否免打扰│     │ · 日志记录  │
                   └──────────┘     └──────────┘
```

| 项目 | 说明 |
|------|------|
| 硬件 | 自复位按键 + 上拉电阻，接入 GPIO |
| 防抖 | 软件消抖 50ms，连续触发间隔 >3s |
| 通知规则 | 白天→TTS"有客到"；夜晚→静默屏显；无人→记录日志 |
| 扩展 | 可区分"外卖"vs"访客"（长按/双击）|

#### ② 温湿度+空调/风扇自动调控

```
传感器轮询 (5s) → 舒适度评估 → 设备控制

FastPath 规则:
  if temp > 28°C and humidity > 70%:   # 闷热
    空调 → 制冷 26°C, 风扇低速
  if temp > 30°C:                       # 酷热
    空调 → 制冷 24°C, 风扇中速
  if temp < 18°C:                       # 偏冷
    空调 → 制热 22°C (如支持)
  if humidity > 80%:                    # 潮湿
    空调 → 除湿模式, 风扇低速
  if humidity < 35%:                    # 干燥
    TTS 提示 → "建议开启加湿器"

SlowPath(LLM) 介入:
  "有点闷热但怕着凉" → Agent 综合判断 → 空调26°C+风扇最低档+风向朝上
```

#### ③ CO₂+空气自动净化

```
传感器轮询 (10s) → CO₂ 分级处理:

  🟢 < 800ppm:  正常, 静默记录
  🟡 800-1200:  开启通风设备(低速), TTS"CO₂偏高,已开启通风"
  🟠 1200-2000: 通风设备(高速) + 净化设备开启, TTS+屏显告警
  🔴 > 2000:    全量通风 + 声光告警 + 建议开窗

空气净化设备选型建议:
  方案A(推荐): 继电器 + 小型USB风扇 + 香薰片
  方案B: 继电器 + 电控喷雾模块
  方案C: 继电器 + 蜂鸣器+风扇(简化)
  → 方案A最直观: 能看到风扇转+闻到香味 = "净化进行中"
```

#### ④ 亮度+灯光自动管理

```
传感器轮询 (3s) → 光照+人体 → 灯光策略:

  有人 + 光照<50 lux  → 开灯
  有人 + 光照<20 lux  → 开灯+全亮
  无人 + 灯亮         → 延迟30s关灯(节能)
  有人 + 光照>300 lux → 保持关灯(自然光充足)

  SlowPath(LLM) 介入:
  "太暗了" → 检查光照 + 检查窗帘状态 → 建议开灯/拉窗帘
```

#### ⑤ 冰箱食材管理与保质期提醒

```
纯软件功能, 无需额外硬件。

用户交互:
  "买了鸡蛋, 5月20号到期, 放在冷藏室"
  → Agent 解析 → add_food("鸡蛋", "2026-05-20", "冷藏", 1)

  "冰箱里还有什么快过期的?"
  → Agent 查询 → "鸡蛋明天过期, 牛奶还有3天"

  "中午能做个什么菜?"
  → Agent 查库存 + LLM 推荐 → "有鸡蛋+番茄+葱, 建议番茄炒蛋"

定时任务:
  每天 08:00 → 检查 3天内过期的食材 → TTS 提醒
  每天 20:00 → 汇总明天过期食材 → Dashboard 推送

数据库设计 (SQLite):
  foods(id, name, category, quantity, unit, purchase_date,
        expiry_date, storage, notes)
  food_logs(id, food_id, action, amount, timestamp)
```

#### ⑥ 外界温度+衣物推荐

```
触发方式:
  ① 用户主动问: "今天穿什么?"
  ② 每日定时推送: 早 07:00 自动建议

工作流:
  1. 获取室外温度（本地 JSON mock / 天气 API）
  2. 读取室内当前温度
  3. LLM 生成建议:

  输入: {"outdoor": "5°C 阴天", "indoor": "22°C"}
  输出: "室外5°C体感较冷, 室内外温差17°C,
        建议穿羽绒服+围巾, 出门注意保暖"

  备用 (LLM 未就绪时):
  if ΔT > 15°C → "温差大, 建议添衣"
  if outdoor < 10°C → "建议穿厚外套"
  if outdoor > 30°C → "建议短袖, 注意防晒"
```

### 3.2 核心组件与内存占用

| 组件 | 选型 | 内存 | 说明 |

## 四、AI 模型与算法设计

### 4.1 模型选型

| 模型 | 参数量 | INT4 大小 | RK3588 NPU 推理速度 | 定位 |
|------|--------|-----------|--------------------|------|
| **Qwen2.5-1.5B-Instruct** | 1.5B | ~900 MB | 10-20 tok/s | **主力**: ReAct 推理/建议/分析 |
| TinyLlama / Qwen2.5-0.5B | 0.5-1.1B | ~350-700 MB | 20-40 tok/s | **备选**: 若 1.5B 延迟过高 |
| BERT-tiny | 4.4M | ~17 MB (INT8) | <5ms | **意图分类/实体抽取** |

**部署方式**：
- **LLM**: RKNN-LLM 工具链量化 → NPU 推理；若兼容问题则 fallback 到 llama.cpp CPU 推理
- **BERT**: RKNN-Toolkit2 导出 INT8 → NPU 推理

### 4.2 三级调用策略（FastPath + MidPath + SlowPath）

**核心设计思想**：LLM 不是万能的，也不是所有场景都需要 LLM。三级策略确保——**80% 场景不走 LLM，走 LLM 的场景一次干完。**

```
                    ┌──────────────────┐
                    │     输入源        │
                    │  传感器 / 按键 / 文本 │
                    └────────┬─────────┘
                             ▼
                    ┌──────────────────┐
                    │  BERT-tiny 分类    │  ← 推理 <5ms
                    │  置信度 + 场景标签  │
                    └────────┬─────────┘
                             │
        ┌────────────────────┼────────────────────┐
        ▼ 置信度 > 90%      ▼ 40-90%            ▼ <40%
   ┌──────────┐      ┌──────────────┐    ┌──────────────┐
   │ FastPath  │      │   MidPath    │    │   SlowPath   │
   │ 规则引擎   │      │  规则先行 +  │    │   LLM 推理   │
   │          │      │  LLM 异步补充  │    │              │
   │ <10ms    │      │  <10ms(规则)  │    │  1-3s       │
   │          │      │  +1-2s(异步)  │    │              │
   │ 温湿度    │      │  "有点热"     │    │ "冰箱里有    │
   │ CO₂阈值   │      │  →先开26°C   │    │  什么快      │
   │ 光照→灯   │      │  后补充"已   │    │  过期了?"    │
   │ 按键通知  │      │  开空调"     │    │              │
   │ 定时任务  │      │              │    │ "今天穿什么?"│
   │          │      │              │    │              │
   │ 覆盖60%  │      │ 覆盖20%      │    │ 覆盖20%      │
   └────┬─────┘      └──────┬───────┘    └──────┬───────┘
        │                   │                    │
        └───────────────────┼────────────────────┘
                            ▼
                   ┌─────────────────┐
                   │   工具编排执行    │
                   │  单步 / 结构化链  │
                   │  并行 / 串行     │
                   └─────────────────┘
```

**三级具体逻辑**：

| 路径 | 触发条件 | 行为 | 用户感知延迟 |
|------|---------|------|-------------|
| **FastPath** | BERT 置信度 > 90% | 规则引擎直接执行，LLM 不参与 | **<10ms** |
| **MidPath** | BERT 置信度 40-90% | **先执行规则动作**（若存在），LLM 异步生成补充回复 | **<10ms** 看到动作，1-2s 后看到回复 |
| **SlowPath** | BERT 置信度 < 40% | 全力走 LLM ReAct，一次输出完整工具链 | **1-3s** |

**MidPath 示例**：

```
用户: "有点热"

① BERT: {"intent": "temperature_adjust", "confidence": 0.65}
② MidPath 触发:
   Step 1 (即时): read_temp() → 32°C → 规则触发 → ac_control(cool, 26)
   Step 2 (异步): LLM 收到 {user: "有点热", action: "已开空调26°C"}
                   → 生成回复 "已开启空调制冷 26°C，一会儿就凉快了"
③ 用户看到: 空调立刻启动 (0.3s)，1s 后 TTS 播报确认
```

**SlowPath 结构化多工具输出**（最大优化）：

```
❌ 标准 ReAct（3轮推理，~6s）：
  Thought: 检查温度 → Action: read_temp()
  Observation: 32°C → Thought: 开空调
  Action: ac_control(cool, 26) → 成功
  Thought: 再开风扇 → Action: set_fan(1)

✅ 结构化输出（1轮推理，~2s）：
  输入 prompt:
    当前环境: temp=32°C, humidity=75%
    用户说: "太热了"
    请一次输出所有需要调用的工具序列:

  输出 (LLM 一次生成):
  [
    {"tool": "ac_control", "params": {"mode": "cool", "temp": 26, "fan_speed": "low"}},
    {"tool": "set_fan", "params": {"speed": 1}}
  ]

  → 系统按序执行 → 结果拼回 → LLM 总结回复
  → 总耗时: 首轮 LLM 1.5s + 设备执行 0.3s + 总结 0.5s = ~2.3s ✅
```

**并行工具执行**（进一步优化多设备场景）：

```python
def execute_plan(tool_calls: list[dict]) -> list:
    """并行执行无依赖的工具调用"""
    results = []
    parallel_group = []
    for call in tool_calls:
        has_dep = any(
            call["tool"] in tool_deps[prev["tool"]]
            for prev in parallel_group
        )
        if has_dep:
            results += run_concurrent(parallel_group)  # 一次性发
            parallel_group = [call]
        else:
            parallel_group.append(call)
    results += run_concurrent(parallel_group)
    return results
```

**举实例对比**：

| 场景 | 标准 ReAct | 结构化+并行 | 提速 |
|------|-----------|------------|------|
| "太热了" (开空调+开风扇) | 3轮 LLM ≈ 6s | 1轮 LLM + 并行执行 ≈ 2.3s | **2.6x** |
| "环境怎么样" (查CO₂+温湿度+光照) | 4轮 LLM ≈ 8s | 1轮 LLM + 3个传感器并行 ≈ 2s | **4x** |
| 按键→通知 | FastPath 直达 | FastPath 直达 | **无需 LLM** |

### 4.3 分层 Agent 架构

不是一个大 Agent 什么都会，而是拆成 **3 个轻量 Agent + 1 个路由器**，让每个 Agent 的 prompt 聚焦、精简。

```

                    │  BERT Router  │  ← 场景分类, <5ms
                    │  意图→子Agent │
                    └──────┬───────┘
           ┌───────────────┼───────────────────┐
           ▼               ▼                   ▼
   ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
   │  环境 Agent   │ │  食材 Agent   │ │  生活 Agent   │
   │              │ │              │ │              │
   │ prompt: 200行│ │ prompt: 180行│ │ prompt: 150行│
   │ 工具: 传感器  │ │ 工具: 食材   │ │ 工具: 天气   │
   │      设备控制 │ │      到期检查 │ │      穿衣建议 │
   │      告警通知 │ │      菜谱推荐 │ │      查询日志 │
   │              │ │              │ │              │
   │ 负责: CO₂/   │ │ 负责: 冰箱   │ │ 负责: 衣物/  │
   │ 温湿度/光照/ │ │ 食材录入/    │ │ 分析/综合报告 │
   │ 空调/风扇/灯 │ │ 提醒/推荐    │ │              │
   └──────────────┘ └──────────────┘ └──────────────┘
```

**收益分析**：

| 指标 | 单体 Agent | 分层 Agent | 改善 |
|------|-----------|-----------|------|
| Prompt 长度 | ~500 行 | ~150-200 行 | **缩短 60%** |
| 首 token 时延 | ~0.8s | ~0.3s | **快 60%** |
| 跨域干扰 | 食材指令可能误触设备 | 路由器隔离，互不干扰 | **无干扰** |
| 可维护性 | 改一个功能需 reload 全部 | 每个 Agent 独立更新 | **拆分维护** |

**注意**：模型只有**一份**（1.5B 共用），不增加内存。差异只在于 system prompt 不同。

### 4.4 上下文窗口管理

**问题**：多轮对话中历史消息堆积，第 10 轮时 prompt 膨胀到 6K+ token，首 token 时延翻倍。

```
轮次 1: [系统指令 + 工具定义 + 用户1 + 回复1]         → ~2K token
轮次 10: [系统指令 + 工具定义 + 9轮历史]              → ~6K token ❌

✅ 优化后:
轮次 10: [系统指令 + 工具定义 + 最近3轮 + 前情摘要]  → ~3K token
```

**实现方案**：

```python
class ConversationManager:
    MAX_HISTORY = 3  # 只保留最近 3 轮

    def build_prompt(self, new_input: str) -> list:
        messages = [self.system_prompt, self.tool_defs]

        if len(self.history) > self.MAX_HISTORY:
            # 将旧轮次压缩为一句摘要
            old = self.history[:-self.MAX_HISTORY]
            summary = self._summarize_old_sessions(old)
            messages.append({"role": "system",
                "content": f"[前情摘要]: {summary}"})

        for session in self.history[-self.MAX_HISTORY:]:
            messages.append(session["user"])
            messages.append(session["assistant"])

        messages.append(new_input)
        return messages

    def _summarize_old_sessions(self, sessions: list) -> str:
        """简单策略: 提取关键事件"""
        key_events = [s["key_action"] for s in sessions if s.get("key_action")]
        return "; ".join(key_events) if key_events else "无关键事件"
```

**收益**：对话持续 50 轮以上，prompt 稳定 ≤3.5K token，首 token 时延不随轮数增长。

- **MidPath (规则先行 + LLM 异步补充)**：覆盖 20% 中等模糊指令，**设备动作零延迟**，LLM 回复异步到达
- **SlowPath (结构化 ReAct)**：覆盖 20% 复杂场景，一次输出完整工具链 + 并行执行，比标准 ReAct 快 2-4x
- **分层 Agent 路由**：按场景分拆 prompt（环境/食材/生活），每个 Agent prompt 缩短 60%

### 5.5 创新五：全端侧低成本方案

- 硬件采购成本 ~70 RMB
- 无云端推理费用，无 API 订阅
- 断网全功能可用，数据零出设备
