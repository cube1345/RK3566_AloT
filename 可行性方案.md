### 2.2 功能详细设计

#### ① 外卖/访客按键提醒

```
用户场景: 外卖员按门外按键 → 室内立刻知道
         屋主不在 → 记录来访时间

工作流:
  ┌──────────┐     ┌──────────┐     ┌──────────┐
  │ GPIO 检测  │────▶ Agent 判断│────▶ 通知执行   │
  │ 按键下降沿 │     │ · 是否在家 │     │ · TTS 播报  │
  │ 防抖处理  │     │ · 时间段   │     │ · 屏显通知  │
  └──────────┘     │ · 是否免打扰│     │ · 日志记录  │
                   └──────────┘     └──────────┘
```

| 项目 | 说明 |
|------|------|
| 硬件 | 自复位按键 + 上拉电阻，接入 GPIO |
| 防抖 | 软件消抖 50ms，连续触发间隔 >3s |
| 通知规则 | 白天→TTS"有客到"；夜晚→静默屏显；无人→记录日志 |
| 扩展 | 可区分"外卖"vs"访客"（长按/双击）|

#### ② 温湿度+空调/风扇自动调控

```
传感器轮询 (5s) → 舒适度评估 → 设备控制

FastPath 规则:
  if temp > 28°C and humidity > 70%:   # 闷热
    空调 → 制冷 26°C, 风扇低速
  if temp > 30°C:                       # 酷热
    空调 → 制冷 24°C, 风扇中速
  if temp < 18°C:                       # 偏冷
    空调 → 制热 22°C (如支持)
  if humidity > 80%:                    # 潮湿
    空调 → 除湿模式, 风扇低速
  if humidity < 35%:                    # 干燥
    TTS 提示 → "建议开启加湿器"

SlowPath(LLM) 介入:
  "有点闷热但怕着凉" → Agent 综合判断 → 空调26°C+风扇最低档+风向朝上
```

#### ③ CO₂+空气自动净化

```
传感器轮询 (10s) → CO₂ 分级处理:

  🟢 < 800ppm:  正常, 静默记录
  🟡 800-1200:  开启通风设备(低速), TTS"CO₂偏高,已开启通风"
  🟠 1200-2000: 通风设备(高速) + 净化设备开启, TTS+屏显告警
  🔴 > 2000:    全量通风 + 声光告警 + 建议开窗

空气净化设备选型建议:
  方案A(推荐): 继电器 + 小型USB风扇 + 香薰片
  方案B: 继电器 + 电控喷雾模块
  方案C: 继电器 + 蜂鸣器+风扇(简化)
  → 方案A最直观: 能看到风扇转+闻到香味 = "净化进行中"
```

#### ④ 亮度+灯光自动管理

```
传感器轮询 (3s) → 光照+人体 → 灯光策略:

  有人 + 光照<50 lux  → 开灯
  有人 + 光照<20 lux  → 开灯+全亮
  无人 + 灯亮         → 延迟30s关灯(节能)
  有人 + 光照>300 lux → 保持关灯(自然光充足)

  SlowPath(LLM) 介入:
  "太暗了" → 检查光照 + 检查窗帘状态 → 建议开灯/拉窗帘
```

#### ⑤ 冰箱食材管理与保质期提醒

```
纯软件功能, 无需额外硬件。

用户交互:
  "买了鸡蛋, 5月20号到期, 放在冷藏室"
  → Agent 解析 → add_food("鸡蛋", "2026-05-20", "冷藏", 1)

  "冰箱里还有什么快过期的?"
  → Agent 查询 → "鸡蛋明天过期, 牛奶还有3天"

  "中午能做个什么菜?"
  → Agent 查库存 + LLM 推荐 → "有鸡蛋+番茄+葱, 建议番茄炒蛋"

定时任务:
  每天 08:00 → 检查 3天内过期的食材 → TTS 提醒
  每天 20:00 → 汇总明天过期食材 → Dashboard 推送

数据库设计 (SQLite):
  foods(id, name, category, quantity, unit, purchase_date,
        expiry_date, storage, notes)
  food_logs(id, food_id, action, amount, timestamp)
```

#### ⑥ 外界温度+衣物推荐

```
触发方式:
  ① 用户主动问: "今天穿什么?"
  ② 每日定时推送: 早 07:00 自动建议

工作流:
  1. 获取室外温度（本地 JSON mock / 天气 API）
  2. 读取室内当前温度
  3. LLM 生成建议:

  输入: {"outdoor": "5°C 阴天", "indoor": "22°C"}
  输出: "室外5°C体感较冷, 室内外温差17°C,
        建议穿羽绒服+围巾, 出门注意保暖"

  备用 (LLM 未就绪时):
  if ΔT > 15°C → "温差大, 建议添衣"
  if outdoor < 10°C → "建议穿厚外套"
  if outdoor > 30°C → "建议短袖, 注意防晒"
```

### 3.5 传感器驱动详解

#### 3.5.1 MH-Z19B CO₂ 传感器

| 项目 | 说明 |
|------|------|
| 接口 | UART (TTL, 3.3V 逻辑) |
| 波特率 | 9600, 8N1 |
| 协议 | 9 字节命令 → 9 字节响应, 校验和: `(0xFF - sum[1:8] + 1) & 0xFF` |
| 命令 | 读 CO₂: `0xFF 0x01 0x86 0x00 0x00 0x00 0x00 0x00 0x79` |
| 接线 | Pi 5 Pin8(TX) → MH-Z19B RX, Pin10(RX) → TX, Pin4(5V)→VCC, Pin6(GND)→GND |
| 代码 | `sensors/co2.py` — pyserial, _ensure_open(), write(cmd) → read(9) → checksum → CO₂=(b[2]<<8)\|b[3] |
| 现象 | Dashboard CO₂ 曲线实时更新; 超阈值触发 TTS"CO₂浓度偏高,已开启通风" / 风扇自动转 |

**关键代码**:
```python
def read(self) -> SensorReading:
    ser = self._ensure_open()
    ser.write(_CMD_READ_CO2)                 # 发送 9 字节命令
    time.sleep(0.01)
    buf = ser.read(9)
    if buf[0] != 0xFF or buf[1] != 0x86:    # 帧头校验
        raise IOError(f"帧头错误: {buf.hex()}")
    co2 = (buf[2] << 8) | buf[3]            # CO₂ ppm
    return SensorReading(value=float(co2), unit="ppm", timestamp=time.time())
```

**备选方案 — SGP30 (I2C CO₂+TVOC)**:

当测试环境没有 MH-Z19B (UART) 时, 可通过 `AGENT_SENSOR_CO2=sgp30` 切换为 SGP30 I2C 传感器:

| 项目 | 说明 |
|------|------|
| 接口 | I2C, 地址 0x58 |
| 输出 | CO₂eq (400-60000 ppm) + TVOC (0-60000 ppb) |
| 协议 | 发送 `0x2003` 初始化 → 预热 15s → 发送 `0x2008` 测量 → 读取 6 字节含 CRC |
| CRC-8 | 同 Sensirion 标准 (0x31, init 0xFF, 与 SHT30 共用代码) |
| 代码 | `sensors/sgp30.py` — 自检, 基线保存/恢复, 湿度补偿 |
| 接线 | Pin3(SDA)→SGP30 SDA, Pin5(SCL)→SCL, Pin1(3.3V)→VCC, Pin6(GND)→GND |

#### 3.5.2 SHT30 温湿度传感器

| 项目 | 说明 |
|------|------|
| 接口 | I2C, 地址 0x44 |
| 协议 | 发送 2 字节测量命令 → 等待 20ms → 读取 6 字节 (温度2+CRC1+湿度2+CRC1) |
| CRC-8 | 多项式 0x31, 初始值 0xFF, 例: `0xBE,0xEF → 0x92` (已验证) |
| 换算 | 温度 = -45 + 175 × raw/65535; 湿度 = 100 × raw/65535 |
| 接线 | Pin3(SDA)→SHT30 SDA, Pin5(SCL)→SCL, Pin1(3.3V)→VCC, Pin6(GND)→GND |
| 代码 | `sensors/temp_humid.py` — smbus2, `i2c_rdwr()` 原始 I2C, 避免 register 地址字节 |
| 现象 | Dashboard 温湿度曲线; 超过 28°C+70% → 自动空调/风扇联动 |

**关键代码**:
```python
def read(self) -> SensorReading:
    bus = self._ensure_open()
    write_msg = smbus2.i2c_msg.write(self._addr, _CMD_MEAS_HIGH_REP)  # 0x2C06
    bus.i2c_rdwr(write_msg)
    time.sleep(0.02)
    read_msg = smbus2.i2c_msg.read(self._addr, 6)
    bus.i2c_rdwr(read_msg)
    buf = bytes(read_msg)
    temp_raw = (buf[0] << 8) | buf[1]
    hum_raw = (buf[3] << 8) | buf[4]
    temp = round(-45.0 + 175.0 * temp_raw / 65535.0, 1)
    humidity = round(100.0 * hum_raw / 65535.0, 1)
    return SensorReading(value=temp, unit="°C", ..., raw={"humidity": humidity})
```

**注意**: 标准 smbus2 的 `read_i2c_block_data()` 会额外发送一个 register 地址字节, 会破坏 SHT30/BH1750 的纯命令协议. 必须使用 `i2c_rdwr()` + `i2c_msg` 进行原始 I2C 读写.

**备选方案 — DHT11 (GPIO 单总线温湿度)**:

当测试环境没有 SHT30 (I2C) 时, 可通过 `AGENT_SENSOR_TEMP=dht11` 切换为 DHT11 GPIO 传感器:

| 项目 | 说明 |
|------|------|
| 接口 | GPIO 单总线 (BCM4, 物理 Pin7) |
| 精度 | 温度 ±2°C (0-50°C), 湿度 ±5% (20-90%) |
| 采样率 | 最大 1Hz (两次读取间隔 ≥1s) |
| 协议 | Host 拉低 20ms → 拉高 30µs → DHT11 响应 (80µs LOW + 80µs HIGH) → 40bits (每 bit: 50µs LOW + 28µs=0 / 70µs=1) |
| 校验 | 8bit 校验和 = (湿度整+湿度小+温度整+温度小) & 0xFF |
| 代码 | `sensors/dht11.py` — gpiod v2 输出→输入双阶段, `time.perf_counter_ns()` 忙等待测量脉冲宽度 |
| 容错 | 读取失败时返回缓存值 + 日志警告, 不中断主循环 |
| 接线 | Pin7(DATA)→DHT11 DATA, Pin1(3.3V)→VCC, Pin6(GND)→GND, DATA 需 4.7kΩ 上拉到 VCC

#### 3.5.3 BH1750 光照传感器

| 项目 | 说明 |
|------|------|
| 接口 | I2C, 地址 0x23 |
| 模式 | 连续高分辨率模式 (命令 `0x10`) |
| 换算 | 照度(lux) = `(buf[0]<<8 | buf[1]) / 1.2` |
| 接线 | Pin3(SDA)→BH1750 SDA, Pin5(SCL)→SCL, Pin1(3.3V)→VCC, Pin6(GND)→GND (与 SHT30 共享 I2C 总线) |
| 代码 | `sensors/light.py` — 发送 `0x10` → 等待 150ms → i2c_rdwr 读 2 字节 |
| 现象 | Dashboard 光照曲线 <50 lux → 灯自动开启; 手遮传感器即触发 |

**关键代码**:
```python
def read(self) -> SensorReading:
    bus = self._ensure_open()
    bus.write_byte(self._addr, _CMD_CONT_H_RES)   # 0x10
    time.sleep(0.15)
    read_msg = smbus2.i2c_msg.read(self._addr, 2)
    bus.i2c_rdwr(read_msg)
    buf = list(read_msg)
    raw = (buf[0] << 8) | buf[1]
    lux = max(0.0, raw / 1.2)
    return SensorReading(value=round(lux, 1), unit="lux", timestamp=time.time())
```

#### 3.5.4 HC-SR501 人体红外传感器

| 项目 | 说明 |
|------|------|
| 接口 | GPIO 输入 (BCM 25, 物理 Pin22) |
| 检测距离 | 3-7 米, 可调延时 ~2-3s 输出脉冲 |
| 防抖 | 软件去抖 0.3s (HC-SR501 脉冲宽度约 2-3s) |
| 降级 | gpiod v2 → gpiod v1 → sysfs (三重 fallback, 兼容所有内核) |
| 接线 | Pin22(DATA)→HC-SR501 OUT, Pin2(5V)→VCC(跳线帽 5V), Pin6(GND)→GND |
| 代码 | `sensors/motion.py` — gpiod 读输入电平, sysfs 兜底 |
| 现象 | Dashboard 人体存在标记; 有人+暗 → 自动开灯; 无人 → 30s 后关灯 |

**三重降级初始化**:
```python
def _init_gpio(self):
    try:
        chip = gpiod.Chip(_CONFIG_GPIO_CHIP)       # /dev/gpiochip0
        if _GPIOD_V2:
            self._req = chip.request_lines(         # gpiod v2 API
                consumer="motion_sensor",
                config={self._pin: gpiod.LineSettings(
                    direction=gpiod.Direction.INPUT)})
        else:
            self._line = chip.get_line(self._pin)   # gpiod v1 API
            self._line.request(consumer="motion_sensor",
                               type=gpiod.LINE_REQ_DIR_IN)
    except Exception:
        self._sysfs_init()                          # sysfs fallback
```

### 3.6 设备驱动详解

#### 3.6.1 风扇 / 灯光 / 空气净化器 — GPIO 继电器控制

三种设备驱动结构完全一致, 区别仅在于 GPIO 引脚和状态变量.

| 设备 | 引脚 (BCM) | 物理 Pin | 状态变量 | 代码文件 |
|------|-----------|---------|---------|---------|
| 灯光 | 17 | Pin11 | state(brightness) | `devices/light.py` |
| 风扇 | 27 | Pin13 | state 0-3 (关/低/中/高) | `devices/fan.py` |
| 净化器 | 10 | Pin19 | level 0-3 | `devices/purifier.py` |

**继电器极性 (`RELAY_ACTIVE_LOW`)**:

大多数国产继电器模块为**低电平触发** (IN=LOW → 继电器吸合, 设备通电). 配置项 `RELAY_ACTIVE_LOW=True` (默认) 处理此极性:

```python
def _apply(self):
    on = self._level > 0
    if RELAY_ACTIVE_LOW:
        gpio_on, gpio_off = gpiod.Value.INACTIVE, gpiod.Value.ACTIVE  # 0=ON, 1=OFF
        raw_on, raw_off = 0, 1
    else:  # 高电平触发 (罕见, 可跳线切换)
        gpio_on, gpio_off = gpiod.Value.ACTIVE, gpiod.Value.INACTIVE  # 1=ON, 0=OFF
        raw_on, raw_off = 1, 0
    if self._req:       # gpiod v2
        self._req.set_value(self._pin, gpio_on if on else gpio_off)
    elif self._line:    # gpiod v1
        self._line.set_value(raw_on if on else raw_off)
```

**gpiod v2/v1 双版本初始化**:
```python
def _init_gpio(self):
    chip = gpiod.Chip(GPIO_CHIP)
    if _GPIOD_V2:
        self._req = chip.request_lines(
            consumer="fan",
            config={self._pin: gpiod.LineSettings(
                direction=gpiod.Direction.OUTPUT,
                output_value=gpiod.Value.INACTIVE)})
    else:
        self._line = chip.get_line(self._pin)
        self._line.request(consumer="fan", type=gpiod.LINE_REQ_DIR_OUT)
```

**接线**: 继电器 VCC → Pi 5 Pin2(5V), GND → Pin6(GND), IN → 对应 BCM GPIO. 继电器 COM/NO 串入设备供电回路.

**可观测现象**:
- 听到继电器吸合的"咔嗒"声
- 风扇扇叶转动 (风感+噪音)
- 灯光亮起
- Dashboard 设备状态卡片实时更新

#### 3.6.2 空调 — IR 红外遥控 (940nm + NEC 协议)

**硬件说明**: 红外发射管为 940nm 波长, **人眼不可见**. 因此空调控制的"可视验证"需结合:
- **红外 LED**: 执行 NEC 协议脉冲序列 → 发送真实空调码 (人眼不可见, 手机摄像头可见)
- **状态指示灯**: 并行接入 GPIO 的可见 LED, IR 发送时同步闪烁 (演示用)
- **Dashboard**: 显示"空调 → 制冷 26°C" 状态变更
- **TTS**: 播报"已开启空调, 设定 26°C"

**NEC 红外协议**:
```
Leader:   9ms ON  +  4.5ms OFF  (载波 38kHz, gpiod 模拟 ON/OFF 时序)
Data:     32 bits LSB-first, 每 bit:
  '0':    560µs ON +  560µs OFF
  '1':    560µs ON + 1690µs OFF
End:      560µs ON
```

**代码** (`devices/ac.py`):
```python
def _send_ir(self):
    code = self._get_code()
    bits = [(code >> i) & 1 for i in range(32)]
    # Leader
    self._set_pin(True); time.sleep(0.009)
    self._set_pin(False); time.sleep(0.0045)
    # 32 bits
    for bit in bits:
        self._set_pin(True); time.sleep(0.00056)
        self._set_pin(False); time.sleep(0.00169 if bit else 0.00056)
    # End
    self._set_pin(True); time.sleep(0.00056)
    self._set_pin(False)
```

**接线**: IR LED 阳极(+) → GPIO BCM22(Pin15) 通过 100Ω 电阻, 阴极(-) → GND. *注意: 940nm 红外光不可见, 调试时可用手机摄像头对准 LED 观察是否发光.*

**可观测现象**:
- 红外 LED (不可见): 手机摄像头可见闪烁
- 并联可见 LED: 快速闪烁 (NEC 协议模式, 肉眼可见)
- Dashboard: "空调 → cool 26°C → 开"
- TTS: "已开启空调, 设定温度 26°C"

#### 3.6.3 TTS 语音播报

| 项目 | 说明 |
|------|------|
| 引擎 | espeak (默认) / Piper-TTS (高质量) 自动检测 |
| 语言 | 中文 (espeak: `zh`, piper: 需中文模型) |
| 降级 | 无 TTS 引擎 → 静默降级到日志记录, 不报错 |
| 代码 | `core/tts.py` — `TTSManager`, speak(text) 异步调用 |
| 现象 | 设备播报中文语音 (如"有客到""CO₂浓度偏高") |

**TTS 原理**: Text-to-Speech 将文本转为波形. espeak 使用共振峰合成 (轻量但机械感强). Piper-TTS 使用神经网络 (高自然度但需 ~300MB 模型). Pi 5 上 espeak 即时可用, Piper 需下载模型后可用.

### 3.7 通信协议架构

```
┌──────────────────────────────────────────────────────────────┐
│                    Raspberry Pi 5 通信总线                      │
│                                                              │
│  GPIO (输出)           GPIO (输入)        I2C           UART │
│  ─────────────────     ────────────     ─────────     ────── │
│  gpiod LineSettings    gpiod get_value   smbus2        pyserial│
│  set_value(0/1)        sysfs fallback   i2c_rdwr      serial  │
│  relay=ON/OFF          debounce 50ms    raw I2C       9600 8N1│
│  LED=on/off            sensor=0/1       no register    cmd→resp│
│  IR=38KHz pulse                         addr byte              │
│                                                              │
│  继电器×3             HC-SR501         SHT30+BH1750   MH-Z19B │

```

| 协议 | 库 | 用途 | 特点 |
|------|----|------|------|
| GPIO 输出 | gpiod | 继电器控制 (灯/风扇/净化器), LED 指示, IR 红外发送 | 电平输出, v2/v1 双版兼容 |
| GPIO 输入 | gpiod / sysfs | 人体红外检测, 按键检测 | 软件防抖, 三重 fallback |
| I2C | smbus2 | SHT30 温湿度, BH1750 光照 | 共享总线, 多设备同地址线, 使用 `i2c_rdwr` 原始模式 |
| UART | pyserial | MH-Z19B CO₂ | 9600 baud, 命令-响应式, 校验和验证 |

export AGENT_PLATFORM=rk3506    # RK3506 开发板

各平台差异 (`config.py`):

| 配置 | pi5 | rk3566 | rk3506 |

### 4.1 模型选型

| 模型 | 参数量 | INT4 大小 | Pi 5 (CPU) 推理速度 | 定位 |
|------|--------|-----------|--------------------|------|

| Qwen2.5-0.5B | 0.5B | ~350 MB | **~17 tok/s** | **备选**: 若需更低延迟 |

**部署方式**：
- **llama.cpp** (CPU) + OpenBLAS 加速 → 利用 Pi 5 4×A76 全核推理
- **模型统一 GGUF 格式**，通过 `llama-cli` 调用，兼容性好，无需 NPU SDK

### 4.2 三级调用策略（FastPath + MidPath + SlowPath）

**核心设计思想**：LLM 不是万能的，也不是所有场景都需要 LLM。三级策略确保——**80% 场景不走 LLM，走 LLM 的场景一次干完。**

```
                    ┌──────────────────┐
                    │     输入源        │
                    │  传感器 / 按键 / 文本 │
                    └────────┬─────────┘
                             ▼
                    ┌──────────────────┐
                    │  BERT-tiny 分类    │  ← 推理 <5ms
                    │  置信度 + 场景标签  │
                    └────────┬─────────┘
                             │
        ┌────────────────────┼────────────────────┐
        ▼ 置信度 > 90%      ▼ 40-90%            ▼ <40%
   ┌──────────┐      ┌──────────────┐    ┌──────────────┐
   │ FastPath  │      │   MidPath    │    │   SlowPath   │
   │ 规则引擎   │      │  规则先行 +  │    │   LLM 推理   │
   │          │      │  LLM 异步补充  │    │              │
   │ <10ms    │      │  <10ms(规则)  │    │  1-3s       │
   │          │      │  +1-2s(异步)  │    │              │
   │ 温湿度    │      │  "有点热"     │    │ "冰箱里有    │
   │ CO₂阈值   │      │  →先开26°C   │    │  什么快      │
   │ 光照→灯   │      │  后补充"已   │    │  过期了?"    │
   │ 按键通知  │      │  开空调"     │    │              │
   │ 定时任务  │      │              │    │ "今天穿什么?"│
   │          │      │              │    │              │
   │ 覆盖60%  │      │ 覆盖20%      │    │ 覆盖20%      │
   └────┬─────┘      └──────┬───────┘    └──────┬───────┘
        │                   │                    │
        └───────────────────┼────────────────────┘
                            ▼
                   ┌─────────────────┐
                   │   工具编排执行    │
                   │  单步 / 结构化链  │
                   │  并行 / 串行     │
                   └─────────────────┘
```

**三级具体逻辑**：

| 路径 | 触发条件 | 行为 | 用户感知延迟 |
|------|---------|------|-------------|
| **FastPath** | BERT 置信度 > 90% | 规则引擎直接执行，LLM 不参与 | **<10ms** |
| **MidPath** | BERT 置信度 40-90% | **先执行规则动作**（若存在），LLM 异步生成补充回复 | **<10ms** 看到动作，1-2s 后看到回复 |
| **SlowPath** | BERT 置信度 < 40% | 全力走 LLM ReAct，一次输出完整工具链 | **1-3s** |

**MidPath 示例**：

```
用户: "有点热"

① BERT: {"intent": "temperature_adjust", "confidence": 0.65}
② MidPath 触发:
   Step 1 (即时): read_temp() → 32°C → 规则触发 → ac_control(cool, 26)
   Step 2 (异步): LLM 收到 {user: "有点热", action: "已开空调26°C"}
                   → 生成回复 "已开启空调制冷 26°C，一会儿就凉快了"
③ 用户看到: 空调立刻启动 (0.3s)，1s 后 TTS 播报确认
```

**SlowPath 结构化多工具输出**（最大优化）：

```
❌ 标准 ReAct（3轮推理，~6s）：
  Thought: 检查温度 → Action: read_temp()
  Observation: 32°C → Thought: 开空调
  Action: ac_control(cool, 26) → 成功
  Thought: 再开风扇 → Action: set_fan(1)

✅ 结构化输出（1轮推理，~2s）：
  输入 prompt:
    当前环境: temp=32°C, humidity=75%
    用户说: "太热了"
    请一次输出所有需要调用的工具序列:

  输出 (LLM 一次生成):
  [
    {"tool": "ac_control", "params": {"mode": "cool", "temp": 26, "fan_speed": "low"}},
    {"tool": "set_fan", "params": {"speed": 1}}
  ]

  → 系统按序执行 → 结果拼回 → LLM 总结回复
  → 总耗时: 首轮 LLM 1.5s + 设备执行 0.3s + 总结 0.5s = ~2.3s ✅
```

**并行工具执行**（进一步优化多设备场景）：

```python
def execute_plan(tool_calls: list[dict]) -> list:
    """并行执行无依赖的工具调用"""
    results = []
    parallel_group = []
    for call in tool_calls:
        has_dep = any(
            call["tool"] in tool_deps[prev["tool"]]
            for prev in parallel_group
        )
        if has_dep:
            results += run_concurrent(parallel_group)  # 一次性发
            parallel_group = [call]
        else:
            parallel_group.append(call)
    results += run_concurrent(parallel_group)
    return results
```

**举实例对比**：

| 场景 | 标准 ReAct | 结构化+并行 | 提速 |
|------|-----------|------------|------|
| "太热了" (开空调+开风扇) | 3轮 LLM ≈ 6s | 1轮 LLM + 并行执行 ≈ 2.3s | **2.6x** |
| "环境怎么样" (查CO₂+温湿度+光照) | 4轮 LLM ≈ 8s | 1轮 LLM + 3个传感器并行 ≈ 2s | **4x** |
| 按键→通知 | FastPath 直达 | FastPath 直达 | **无需 LLM** |

### 4.3 分层 Agent 架构

不是一个大 Agent 什么都会，而是拆成 **3 个轻量 Agent + 1 个路由器**，让每个 Agent 的 prompt 聚焦、精简。

```
                    ┌──────────────┐
                    │  BERT Router  │  ← 场景分类, <5ms
                    │  意图→子Agent │
                    └──────┬───────┘
           ┌───────────────┼───────────────────┐
           ▼               ▼                   ▼
   ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
   │  环境 Agent   │ │  食材 Agent   │ │  生活 Agent   │
   │              │ │              │ │              │
   │ prompt: 200行│ │ prompt: 180行│ │ prompt: 150行│
   │ 工具: 传感器  │ │ 工具: 食材   │ │ 工具: 天气   │
   │      设备控制 │ │      到期检查 │ │      穿衣建议 │
   │      告警通知 │ │      菜谱推荐 │ │      查询日志 │
   │              │ │              │ │              │
   │ 负责: CO₂/   │ │ 负责: 冰箱   │ │ 负责: 衣物/  │
   │ 温湿度/光照/ │ │ 食材录入/    │ │ 分析/综合报告 │
   │ 空调/风扇/灯 │ │ 提醒/推荐    │ │              │
   └──────────────┘ └──────────────┘ └──────────────┘
```

**收益分析**：

| 指标 | 单体 Agent | 分层 Agent | 改善 |
|------|-----------|-----------|------|
| Prompt 长度 | ~500 行 | ~150-200 行 | **缩短 60%** |
| 首 token 时延 | ~0.8s | ~0.3s | **快 60%** |
| 跨域干扰 | 食材指令可能误触设备 | 路由器隔离，互不干扰 | **无干扰** |
| 可维护性 | 改一个功能需 reload 全部 | 每个 Agent 独立更新 | **拆分维护** |

**注意**：模型只有**一份**（1.5B 共用），不增加内存。差异只在于 system prompt 不同。

### 4.4 上下文窗口管理

**问题**：多轮对话中历史消息堆积，第 10 轮时 prompt 膨胀到 6K+ token，首 token 时延翻倍。

```
轮次 1: [系统指令 + 工具定义 + 用户1 + 回复1]         → ~2K token
轮次 10: [系统指令 + 工具定义 + 9轮历史]              → ~6K token ❌

✅ 优化后:
轮次 10: [系统指令 + 工具定义 + 最近3轮 + 前情摘要]  → ~3K token
```

**实现方案**：

```python
class ConversationManager:
    MAX_HISTORY = 3  # 只保留最近 3 轮

    def build_prompt(self, new_input: str) -> list:
        messages = [self.system_prompt, self.tool_defs]

        if len(self.history) > self.MAX_HISTORY:
            # 将旧轮次压缩为一句摘要
            old = self.history[:-self.MAX_HISTORY]
            summary = self._summarize_old_sessions(old)
            messages.append({"role": "system",
                "content": f"[前情摘要]: {summary}"})

        for session in self.history[-self.MAX_HISTORY:]:
            messages.append(session["user"])
            messages.append(session["assistant"])

        messages.append(new_input)
        return messages

    def _summarize_old_sessions(self, sessions: list) -> str:
        """简单策略: 提取关键事件"""
        key_events = [s["key_action"] for s in sessions if s.get("key_action")]
        return "; ".join(key_events) if key_events else "无关键事件"
```

**收益**：对话持续 50 轮以上，prompt 稳定 ≤3.5K token，首 token 时延不随轮数增长。

## 八、风险与应对

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 无 NPU, CPU 推理延迟较高 | — | 中 | ~10 tok/s Qwen2.5-1.5B, 实测可用; 备选 0.5B 提速 |
| 单人工期压力大 | 高 | 高 | **砍到 P0 再砍**: 温湿度+CO₂+灯光 核心3闭环 |
| 传感器 I2C 掉线 | 中 | 中 | 看门狗 + 自动重连 + 降级策略 |
| 红外码找不到对应空调型号 | 低 | 中 | 学习型遥控器录制; 或 Broadlink RM 集成 |
| 食材信息手工录入频次低 | 中 | 低 | 设计简洁的录入交互, 一次录入永久跟踪 |
| 演示时传感器不灵敏 | 低 | 高 | 预录 90% + 现场 10% 交互 |
| LLM 结构化 JSON 输出格式异常 | 中 | 中 | try/except 兜底 + 重试 1 次 + 降级到规则引擎 |
| 上下文摘要丢失关键信息 | 低 | 中 | 摘要策略保留 key_action 字段，不做纯 LLM 摘要 |

### 14.4 核心模块详解

#### AgentOrchestrator (`core/agent.py`)

系统主控制器, 负责全生命周期管理。

    def handle_command(self, text, history=None) -> dict:

**关键设计决策:**
- 传感器轮询和规则引擎在同一协程: 避免竞争条件
- Flask 跑在 daemon thread: 无需额外进程, 主进程退出自动清理
- LLM 文件不存在则静默降级: Mock 模式下无模型也能完整运行

#### FastPath 规则引擎 (`core/fastpath.py`)

传感器阈值触发, <10ms 延迟, 零 LLM 开销。

```python
# 预置 6 条规则:
#   ① 温度 >28°C + 湿度 >70% → 空调制冷 + 风扇
#   ② 温度 >30°C → 空调强力制冷
#   ③ 温度 <18°C → 空调制热
#   ④ 湿度 >80% → 除湿模式
#   ⑤ CO₂ 800-1200 → 通风低速
#   ⑥ CO₂ 1200-2000 → 通风高速 + 风扇
#   ⑦ CO₂ >2000 → 紧急告警 + 全速通风
#   ⑧ 光照 <20 lux + 有人 → 全亮
#   ⑨ 光照 20-50 lux + 有人 → 半亮
#   ⑩ 光照 >300 lux + 灯亮 → 关灯
#   ⑪ 定时 07:00 → 穿衣建议
#   ⑫ 定时 08:00 → 食材到期检查
```

每条规则有冷却时间 (cooldown), 防止频繁触发。

#### 分层 Agent Prompts

每个 Agent 独立 system prompt, 包含角色定义 + 工具列表 + 输出规则 + 示例:

- **environment_agent.py** (12 个工具): 环境控制, CO₂/温湿度/光照/设备
- **food_agent.py** (6 个工具): 食材管理, 自然语言→结构化入库
- **life_agent.py** (9 个工具): 生活建议, 天气/穿衣/日报/贴士

Prompt 设计原则:
1. **JSON-only 硬约束**: "只输出 JSON 数组格式的工具调用, 不要输出对话文字"
2. **具体示例**: 每条规则附带输入→输出示例, 覆盖典型场景
3. **不确认不废话**: 不输出"好的,我来帮您..."等确认语句
4. **空场景处理**: "不需要调用工具就输出 []"

### 14.8 已知限制

| 限制 | 描述 | 影响 | 缓解措施 |
|------|------|------|----------|
| DHT11 GPIO 时序 | 忙等待依赖 Python 解释器调度, 非 RTOS | 高负载下可能位采样偏移 | 设 1Hz 轮询上限, 读取失败回退缓存值 |
| SGP30 预热 | ~15s 内输出固定 400ppm/0ppb | 上电后 15s CO₂ 数据不可用 | `_warmup_done` 标记, 期间不上报告警 |
| IR LED 不可见 | 940nm 红外肉眼不可见 | 调试困难, 无法目视确认发射 | 用手机摄像头 (可见紫色光斑) 或逻辑分析仪 |
| 单用户设计 | 无用户认证/多用户隔离 | 不适合多用户家庭 | 初赛演示无需此功能 |
| 无 MQTT | 设备通信同进程直调 | 难以扩展到分布式/多房间 | Phase 4 P1 任务, 视进度决定 |
| LLM 中文分词 | Qwen2.5-1.5B 中文能力有限 | 复杂语义理解可能偏差 | 规则引擎覆盖 80% 高频场景, LLM 只兜底 |
| 红外码库单一 | 仅存储一套空调 IR 码 | 不匹配其他空调型号需学习 | `devices/ac.py` 定义为常量, 可替换 |
| Mock vs 真实一致性 | Mock 传感器时序/噪声与现实不同 | Mock 通过的真实接线可能有偏差 | 压力测试+边界值覆盖, 硬件接线后逐场景验证 |

### 14.9 常见问题

**Q: Mock 模式下没有 LLM 模型也能运行吗?**
A: 可以。`AGENT_LLM` 指向的文件不存在时, AgentOrchestrator._init_llm() 告警但不阻塞启动, CommandHandler 降级为规则引擎 + 正则兜底。所有 90 个测试和 7 大功能在 Mock 模式均可完整走通。

**Q: 如何切换传感器型号?**
A: 设置环境变量: `export AGENT_SENSOR_CO2=sgp30 AGENT_SENSOR_TEMP=dht11`, 然后以 `AGENT_MOCK=0` 启动。系统自动选择 `sensors/sgp30.py` 而非 `sensors/co2.py`, 选择 `sensors/dht11.py` 而非 `sensors/temp_humid.py`。

**Q: 如何切换运行平台?**
A: 设置 `export AGENT_PLATFORM=rk3566` (或 `rk3506`), 配置自动切换 GPIO 引脚映射表、UART 默认路径和 I2C 总线号。

**Q: 有 LLM 但没有传感器会怎样?**
A: 全部功能正常。Mock 传感器提供模拟数据, LLM 正常推理, Dashboard 正常显示。适合纯软件演示。

**Q: 为什么风扇/灯光是 ON/OFF 而非 PWM?**
A: 开发板(包括 Pi 5 和 RK 系列)的 GPIO 是数字 IO, 非专用 PWM 引脚。软件 PWM 不精准且占用 CPU。风机通过继电器控制启停, 灯光通过继电器控制开关, 亮度用 GPIO 控制的灯具本身支持调光。
