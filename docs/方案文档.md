## 一、场景选择与痛点分析

### 1.1 场景定位：家庭智能家居管家

本项目面向**家庭场景**下的全屋环境管理需求，构建一个运行在树莓派5上的端侧AI管家，无需联网即可自主决策、执行设备控制并与用户自然对话。

### 1.2 核心痛点

| 痛点 | 现有方案局限 | Agent ZX 方案 |
|------|-------------|--------------|
| **云依赖** | 市面智能家居依赖云端AI，延迟高、断网即瘫痪 | 100% 端侧推理，离线可用 |
| **隐私泄露** | 家庭传感器数据上传云端 | 数据全本地处理，不上传 |
| **被动响应** | 智能音箱等需用户主动唤醒 | AI每30s主动评估传感器，自主决策 |
| **单设备孤岛** | 各设备独立控制，无联动 | 多传感器融合 → 整屋联动 |
| **缺乏学习** | 固化的自动化规则，千人一面 | 偏好学习积累用户习惯，个性化适应 |

### 1.3 创新点总结

1. **CoT思考链推理** — LLM输出结构化为「观察→分析→决策」三步, 推理过程可追溯可审计, 不再黑箱输出
2. **LLM驱动主动决策** — 不同于传统规则引擎的被动响应，LLM每30s评估全屋传感器快照，自主发起操作
3. **三级决策架构** — FastPath(紧急安全网<5ms) + AIBrain(LLM决策30s周期) + MidPath(自然语言补充)，兼顾实时性与智能性
4. **模糊指令主动追问** — 指代不明/动作不完整/含义模糊等4种模式智能追问, 提供选项按钮, 多轮对话累积上下文
5. **场景识别联动** — "晚安"一词触发关灯+空调26°C+风扇关+免打扰，6个预定义场景+时间自动触发
6. **AI主动反问机制** — AI不确定时反问用户（如"CO₂偏高但你在做饭，要通风吗？"），Web 5s轮询, 用户选择后执行
7. **用户画像引擎** — 5维度行为追踪(活跃时段+Agent使用+指令长度+温度偏好+场景频次), 规律自动发现, 个性化响应注入LLM
8. **知识驱动决策** — 决策前检索历史数据（同时段对比、日均趋势、传感器关联），实现RAG增强推理
9. **端侧全栈** — 传感器驱动→数据库→LLM推理→Web Dashboard，全部运行在树莓派5上, GPIO/I2C/UART全协议覆盖

### 2.2 决策架构 (v5.3)

| 路径 | 延迟 | 触发条件 | LLM参与 | 职责 |
|------|------|---------|---------|------|
| **FastPath** | <5ms | temp>35°C / CO₂>2000ppm | 否 | 硬件安全底线，紧急兜底 |
| **AIBrain** | 1-3s | 每30s周期评估 | 是 | 主动决策核心: 综合传感器+画像+知识上下文 |

### 2.2.1 CoT 思考链 (v5.2)

LLM 输出不再黑箱，结构化为三个步骤:

```
观察: <理解用户指令+当前环境状态>
分析: <推理用户真实意图，结合传感器数据>
决策: [{"tool":"...","params":{}}] 或 []
```

- **解析**: 支持8种JSON回退格式确保鲁棒性
- **回复**: 优先提取「分析」作为自然语言回复，而非裸工具结果
- **注入**: 用户画像摘要 + 对话历史压缩注入 System Prompt
- **FEW-SHOT**: 3个Agent各含CoT格式领域示例

### 2.2.2 用户画像引擎 (v5.2)

```
行为追踪(user_behaviors) → 5维度画像(user_profile)
  ├─ active_hours:  各时段活跃度直方图
  ├─ agent_usage:   Agent使用频率分布
  ├─ command_length: 指令长度均值/p50/p90
  ├─ temp_preference: 温度偏好数值
  └─ scene_frequency: 场景触发频次

规律自动发现: SQL按时段+行为类型跨天聚合 → 置信度≥30% → 自动创建routine
个性化注入: ~50字画像摘要注入LLM System Prompt
```

### 2.2.3 模糊指令追问 (v5.2)

4种模糊模式逐层检测:

| 模式 | 正则示例 | 追问示例 |
|------|---------|---------|
| 指代不明 | `/把它(?:关|开|调)/` | "您是指灯、风扇还是空调？" |
| 动作不完整 | `/^开一下/` | "想开哪个设备呢？" |
| 含义模糊 | `/有点不舒服/` | "是温度不合适还是空气质量不好？" |
| 上下文依赖 | `/^(然后|还有|接着)$/` | "还需要我做什么呢？" |

返回 `{type:"clarification", text, options:[...]}`，Dashboard 显示选项按钮，用户回答后合并上下文重新路由。

## 三、AI模型与算法说明

### 3.1 模型选型

| 项目 | 规格 |
|------|------|
| **模型** | Qwen2.5-1.5B-Instruct Q4_K_M GGUF |
| **参数量** | 1.5B (量化后 ~941MB) |
| **推理引擎** | llama.cpp (C++后端) |
| **上下文窗口** | 4096 tokens |
| **推理速度** | ~15-20 tok/s (Pi 5 CPU, n_threads=4全核, n_batch=512) |
| **Temperature** | 0.0 (确定性输出，确保设备控制一致性) |
| **最大输出** | 512 tokens |
| **内存占用** | ~1.2GB (含KV cache) |

### 3.2 量化策略

- **GGUF Q4_K_M量化**: 4-bit量化 + K-Quant混合精度，关键层保留更高精度
- **压缩比**: 原始FP16 ~3GB → 量化后 941MB，压缩比 ~3.2:1
- **精度影响**: 在结构化JSON输出任务上无明显退化（temperature=0.0确保确定性）

### 3.3 推理优化

| 优化措施 | 方法 | 效果 |
|----------|------|------|
| **多线程推理** | n_threads=4 (Pi5全核), n_batch=512 | 推理速度 10→15-20 tok/s |
| **CoT格式统一** | AIBrain+CommandHandler双引擎统一观察→分析→决策 | 一致可追溯推理, 解析复用 |
| **Prompt缓存** | 缓存ProfileEngine实例, 避免重复创建 | 减少每次调用开销 |
| **Context限制** | 只注入最近3轮决策+高置信度偏好（>0.3） | Token消耗 ~350/次 |
| **Prompt精简** | System prompt压缩, 工具定义紧凑 | 减少KV cache开销 |
| **决策间隔** | AI 30s评估一次，非连续推理 | 避免CPU持续满载 |
| **ReAct限制** | 最多2轮迭代，避免无限推理 | 单次指令最长4s |
| **双后端设计** | CPU (llama.cpp) + 可选NPU (RKNN for RK3566) | 硬件适配灵活 |

### 3.4 关键算法

#### 3.4.1 CoT思考链推理 (v5.2)

```
LLM输出结构化为三个步骤:
  观察: <理解用户指令+当前环境状态, 读传感器>
  分析: <推理用户真实意图，结合传感器数据，判断是否需要行动>
  决策: [{"tool":"...","params":{}}] 或 []

解析: 支持8种JSON回退格式确保鲁棒性（含markdown代码块、裸JSON、混入文本等）
回复: 优先提取「分析」字段作为自然语言回复，而非裸工具结果
注入: 用户画像摘要(~50字) + 对话历史压缩 + 各Agent CoT few-shot示例注入System Prompt
ReAct: 最多2轮迭代，每轮保留观察/分析文本作为推理轨迹注入下一轮
```

#### 3.4.2 用户画像引擎 (v5.2)

```
行为追踪(user_behaviors) → 5维度画像(user_profile)
  ├─ active_hours:    各时段活跃度直方图
  ├─ agent_usage:     Agent使用频率分布
  ├─ command_length:  指令长度均值/p50/p90
  ├─ temp_preference: 温度偏好数值
  └─ scene_frequency: 场景触发频次

规律自动发现: SQL按时段+行为类型跨天聚合 → 置信度≥30% → 自动创建routine
个性化注入: ~50字画像摘要注入LLM System Prompt
```

#### 3.4.3 AI决策 (evaluate)

```
输入: 传感器快照 {co2, temperature, humidity, light, person_present, k210}
     + 历史知识上下文 (同时段对比 ±15%偏差时注入)
     + 最近3次操作 (防重复)
     + 用户偏好 + 用户画像摘要 (高置信度)
     + 当前时段建议 (6-9/11-13/17-19/21-23时)

LLM推理 → 结构化解析:
  [action]   EXPLANATION: <原因> + TOOLS: [{tool, params}]
  [question] QUESTION: <问题> + OPTIONS: [选项] + PENDING: [待执行工具]
  [proactive] TEXT: <时段主动建议>
  [none]     无需操作

输出: {type, tool_chain, explanation}
```

#### 3.4.4 异常检测 (detect_anomaly)

```
方法: z-score滑动窗口检测
参数: 窗口=10, 阈值=3.0σ
逻辑: |new_val - μ| / σ > 3.0 → 跳过自动操作，标记异常
```

#### 3.4.5 偏好学习 (learn_preference)

```
触发条件: 用户60s内覆盖AI决策
方法: 置信度递增 (每次+0.15, 上限1.0)
存储: user_prefs表 (key, value, confidence, count)
注入: 高置信度(>0.3)偏好注入LLM context
```

#### 3.4.6 知识驱动决策

```
query_sensor_same_hour(sensor, days=7):
  → strftime('%H')匹配 → 同时段历史对比
  → 偏差>15%时注入LLM prompt
  示例: "CO₂当前850, 过去7天同时段均值780 (偏高70)"

query_sensor_hourly_profile(sensor, days=7):
  → GROUP BY hour → AVG/MIN/MAX
  → 用于日洞察生成

query_sensor_correlation(sensor_a, sensor_b, hours=24):
  → 分钟桶JOIN → 传感器关联分析
```

### 4.1 工具注册体系 (ToolRegistry)

```python
@registry.register(description="设置空调模式、温度、风速")
def ac_control(mode: str, temp: float, fan_speed: str) -> str:
    ...

# 动态生成工具描述块注入LLM prompt
tool_block = registry.get_prompt_block()
# → "1. ac_control(mode, temp, fan_speed) — 设置空调模式、温度、风速\n"
```

### 4.4 ReAct多步推理 (含CoT)

复杂指令触发ReAct循环，每轮输出CoT格式（观察→分析→决策），轨迹注入下一轮：

```
用户: "家里太闷了，帮我检查一下所有设备"
  → Iter1 CoT:
    观察: 用户反馈闷，当前CO₂ 1100ppm偏高，湿度75%
    分析: 需降温除湿通风，综合处理
    决策: [ac_control(cool,26), set_fan(2)]
  → 执行后环境反馈, Iter2 CoT:
    观察: 空调26°C风扇2档运转中，空气净化未开
    分析: 补充开启净化器进一步改善空气
    决策: [set_air_purifier(1)]
  → 最终: 3个工具执行, 提取分析字段作为自然语言回复
```

## 五、性能优化措施

### 5.1 算力适配 (≤6 TOPS, 4GB内存)

| 指标 | 实际值 | 约束 |
|------|--------|------|
| LLM内存 | ~1.2GB | <4GB |
| 推理延迟 | 1-3s/次 | 非阻塞(异步) |
| CPU占用 | ~25% (单核推理) | 留足传感器轮询 |
| Token/决策 | ~350 | 上下文4096安全 |
| 磁盘占用 | ~950MB (模型) + ~5MB (代码+DB) | <8GB |

### 5.2 关键优化技术

1. **三级决策分层** — 紧急场景跳过LLM，规则<5ms响应
2. **CoT结构化输出** — AIBrain+CommandHandler统一观察→分析→决策，8种回退解析，确定性高
3. **多线程推理** — llama.cpp n_threads=4 (Pi5全核) + n_batch=512，推理速度 ~15-20 tok/s
4. **异步非阻塞** — AI决策为async task，传感器轮询不等待LLM
5. **Prompt工程** — CoT few-shot示例+用户画像摘要指导输出格式，ProfileEngine实例缓存
6. **SQLite WAL模式** — 写操作不阻塞读，支持并发
7. **DHT11缓存** — 读取间隔1.5s，失败时返回上次有效值
8. **SGP30基线复用** — 支持基线保存/恢复，减少预热时间
