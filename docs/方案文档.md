## 一、场景选择与痛点分析

### 1.1 场景定位：家庭智能家居管家

本项目面向**家庭场景**下的全屋环境管理需求，构建一个运行在树莓派5上的端侧AI管家，无需联网即可自主决策、执行设备控制并与用户自然对话。

### 1.2 核心痛点

| 痛点 | 现有方案局限 | Agent ZX 方案 |
|------|-------------|--------------|
| **云依赖** | 市面智能家居依赖云端AI，延迟高、断网即瘫痪 | 100% 端侧推理，离线可用 |
| **隐私泄露** | 家庭传感器数据上传云端 | 数据全本地处理，不上传 |
| **被动响应** | 智能音箱等需用户主动唤醒 | AI每30s主动评估传感器，自主决策 |
| **单设备孤岛** | 各设备独立控制，无联动 | 多传感器融合 → 整屋联动 |
| **缺乏学习** | 固化的自动化规则，千人一面 | 偏好学习积累用户习惯，个性化适应 |

### 1.3 创新点总结

1. **LLM驱动主动决策** — 不同于传统规则引擎的被动响应，LLM每30s评估全屋传感器快照，自主发起操作
2. **三级决策架构** — FastPath(紧急安全网<5ms) + AIBrain(LLM决策30s周期) + MidPath(自然语言补充)，兼顾实时性与智能性
3. **场景识别联动** — "晚安"一词触发关灯+空调26°C+风扇关+免打扰，6个预定义场景覆盖日常高频操作
4. **主动反问机制** — AI不确定时反问用户（如"CO₂偏高但你在做饭，要通风吗？"），而非盲目执行
5. **知识驱动决策** — 决策前检索历史数据（同时段对比、日均趋势、传感器关联），实现RAG增强推理
6. **端侧全栈** — 传感器驱动→数据库→LLM推理→Web Dashboard，全部运行在≤6 TOPS / 4GB内存的树莓派上

### 2.2 三级决策架构

| 路径 | 延迟 | 触发条件 | LLM参与 | 职责 |
|------|------|---------|---------|------|
| **FastPath** | <5ms | temp>35°C / CO₂>2000ppm | 否 | 硬件安全底线，紧急兜底 |
| **AIBrain** | 1-3s | 每30s周期评估 | 是 | 主动决策核心：温度/CO₂/光照综合判断 |
| **MidPath** | 异步 | AIBrain执行后 | 是 | 自然语言解释已执行的操作 |

### 2.3 数据流

```
传感器(SGP30/DHT11/BH1750/HC-SR501)  →  [轮询 1-10s]
  │
  ├─→ FastPath 安全网 ──── 紧急触发 → 设备执行
  │
  ├─→ AIBrain.evaluate() ─→ LLM推理(含知识上下文)
  │     │
  │     ├─ action   → ToolRegistry执行 → MidPath说明 → Dashboard
  │     ├─ question → pending_question → Web 5s轮询 → 用户答案 → 执行/忽略
  │     └─ none     → 日志记录
  │
  └─→ 异常检测(z-score) ─→ 跳变 → 跳过自动操作
```

## 三、AI模型与算法说明

### 3.1 模型选型

| 项目 | 规格 |
|------|------|
| **模型** | Qwen2.5-1.5B-Instruct Q4_K_M GGUF |
| **参数量** | 1.5B (量化后 ~941MB) |
| **推理引擎** | llama.cpp (C++后端) |
| **上下文窗口** | 4096 tokens |
| **推理速度** | ~10 tok/s (Pi 5 CPU) |
| **Temperature** | 0.0 (确定性输出，确保设备控制一致性) |
| **最大输出** | 512 tokens |
| **内存占用** | ~1.2GB (含KV cache) |

### 3.2 量化策略

- **GGUF Q4_K_M量化**: 4-bit量化 + K-Quant混合精度，关键层保留更高精度
- **压缩比**: 原始FP16 ~3GB → 量化后 941MB，压缩比 ~3.2:1
- **精度影响**: 在结构化JSON输出任务上无明显退化（temperature=0.0确保确定性）

### 3.3 推理优化

| 优化措施 | 方法 | 效果 |
|----------|------|------|
| **Context限制** | 只注入最近3轮决策+高置信度偏好（>0.3） | Token消耗 ~350/次 |
| **Prompt精简** | System prompt压缩至250字，工具定义紧凑 | 减少KV cache开销 |
| **决策间隔** | AI 30s评估一次，非连续推理 | 避免CPU持续满载 |
| **ReAct限制** | 最多2轮迭代，避免无限推理 | 单次指令最长6s |
| **双后端设计** | CPU (llama.cpp) + 可选NPU (RKNN for RK3566) | 硬件适配灵活 |

### 3.4 关键算法

#### 3.4.1 AI决策 (evaluate)

```
输入: 传感器快照 {co2, temperature, humidity, light, person_present}
     + 历史知识上下文 (同时段对比 ±15%偏差时注入)
     + 最近3次操作 (防重复)
     + 用户偏好 (高置信度)

LLM推理 → 结构化解析:
  [action]   EXPLANATION: <原因> + TOOLS: [{tool, params}]
  [question] QUESTION: <问题> + OPTIONS: [选项] + PENDING: [待执行工具]
  [none]     无需操作

输出: {type, tool_chain, explanation}
```

#### 3.4.2 异常检测 (detect_anomaly)

```
方法: z-score滑动窗口检测
参数: 窗口=10, 阈值=3.0σ
逻辑: |new_val - μ| / σ > 3.0 → 跳过自动操作，标记异常
```

#### 3.4.3 偏好学习 (learn_preference)

```
触发条件: 用户60s内覆盖AI决策
方法: 置信度递增 (每次+0.15, 上限1.0)
存储: user_prefs表 (key, value, confidence, count)
注入: 高置信度(>0.3)偏好注入LLM context
```

#### 3.4.4 知识驱动决策

```
query_sensor_same_hour(sensor, days=7):
  → strftime('%H')匹配 → 同时段历史对比
  → 偏差>15%时注入LLM prompt
  示例: "CO₂当前850, 过去7天同时段均值780 (偏高70)"

query_sensor_hourly_profile(sensor, days=7):
  → GROUP BY hour → AVG/MIN/MAX
  → 用于日洞察生成

query_sensor_correlation(sensor_a, sensor_b, hours=24):
  → 分钟桶JOIN → 传感器关联分析
```

## 四、工具函数设计与调用逻辑

### 4.1 工具注册体系 (ToolRegistry)

基于装饰器的工具注册，支持并行执行和参数自动校验：

```python
@registry.register(description="设置空调模式、温度、风速")
def ac_control(mode: str, temp: float, fan_speed: str) -> str:
    ...

# 动态生成工具描述块注入LLM prompt
tool_block = registry.get_prompt_block()
# → "1. ac_control(mode, temp, fan_speed) — 设置空调模式、温度、风速\n"
```

### 4.2 工具清单 (12个核心工具)

| 工具 | 参数 | 功能 |
|------|------|------|
| `ac_control` | mode, temp, fan_speed | 空调控制(制冷/制热/除湿/关) |
| `set_fan` | speed (0-3) | 风扇调速 |
| `set_light` | state, brightness | 灯光开关+亮度 |
| `set_air_purifier` | level (0-3) | 空气净化器档位 |
| `tts` | text | 语音播报 |
| `notify_display` | title, body | Dashboard通知推送 |
| `get_weather` | — | 获取室外天气数据 |
| `add_food` | name, expiry_date, storage, quantity, unit | 添加食材 |
| `list_foods` | expiring_days? | 查看食材列表 |
| `remove_food` | id | 删除食材 |
| `get_dress_advice` | — | 获取穿衣建议 |
| `get_sensor_value` | sensor_name | 读取传感器当前值 |

### 4.3 场景工具链 (SceneEngine)

6个预定义场景，每个包含多设备联动工具链：

| 场景 | 触发词 | 工具链 |
|------|--------|--------|
| 睡觉 | 晚安/睡了/睡觉 | 关灯 + 空调26°C低风 + 风扇关 |
| 出门 | 出门/走了/拜拜 | 关灯 + 关空调 + 关风扇 + 关净化器 |
| 回家 | 回来了/到家 | 空调26°C自动风 |
| 观影 | 看电影/追剧 | 灯光50% + 空调25°C低风 |
| 起床 | 早安/早上好 | 灯光100% + 关空调 |
| 烹饪 | 做饭/炒菜 | 净化器2档 |

### 4.4 ReAct多步推理

复杂指令（>15字或含疑问词）触发ReAct循环：

```
用户: "家里太闷了，帮我检查一下所有设备"
  → Iter1: [ac_control(cool,26), set_fan(2)]
  → 观察: "空调26°C, 风扇2档"
  → Iter2: [set_air_purifier(1)]  (补充净化)
  → 最终: 3个工具执行, MidPath生成自然语言说明
```

### 4.5 指令路由

3个领域Agent按关键词分发：

| Agent | 关键词 | 负责领域 |
|-------|--------|---------|
| environment | 温度/CO₂/灯光/空调/风扇/空气 | 环境调控 |
| food | 买/食材/冰箱/过期/菜谱 | 食材管理 |
| life | 天气/穿衣/提醒/日报 | 生活助手 |

## 五、性能优化措施

### 5.1 算力适配 (≤6 TOPS, 4GB内存)

| 指标 | 实际值 | 约束 |
|------|--------|------|
| LLM内存 | ~1.2GB | <4GB |
| 推理延�� | 1-3s/次 | 非阻塞(异步) |
| CPU占用 | ~25% (单核推理) | 留足传感器轮询 |
| Token/决策 | ~350 | 上下文4096安全 |
| 磁盘占用 | ~950MB (模型) + ~5MB (代码+DB) | <8GB |

### 5.2 关键优化技术

1. **三级决策分层** — 紧急场景跳过LLM，规则<5ms响应
2. **输出结构化约束** — LLM只输出TOOLS JSON数组，不做自由对话（确定性高、解析快）
3. **异步非阻塞** — AI决策为async task，传感器轮询不等待LLM
4. **Prompt工程** — Few-shot示例指导输出格式，减少重试
5. **SQLite WAL模式** — 写操作不阻塞读，支持并发
6. **DHT11缓存** — 读取间隔1.5s，失败时返回上次有效值
7. **SGP30基线复用** — 支持基线保存/恢复，减少预热时间

### 5.3 延迟分析

| 操作 | 延迟 | 说明 |
|------|------|------|
| 传感器读取 | 1-20ms | SGP30 12ms / DHT11 1.5s(含间隔) |
| FastPath触发 | <5ms | 纯Python规则 |
| AIBrain决策 | 1-3s | LLM推理(~35 tokens × 100ms/tok) |
| ReAct多轮 | 2-6s | 最多2轮 |
| 工具执行 | <50ms | 本地GPIO/I2C操作 |
| 场景识别 | <1ms | 内存词典匹配 |

## 七、可扩展性与实用性评估

### 7.1 硬件兼容性

| 平台 | 内存 | NPU | 适配状态 |
|------|------|-----|---------|
| Raspberry Pi 5 | 8GB | 无 | 默认目标, CPU推理 |
| RK3566 | 4GB | 0.8 TOPS | 已适配(RKNN后端), 引脚映射就绪 |
| RK3506 | 2GB | 1 TOPS | 引脚映射就绪, 模型需进一步裁剪 |

### 7.2 场景扩展

- **新增场景**: 只需在 `config.py` 的 `SCENE_TRIGGERS` 添加条目（关键词+工具链+回复）
- **新增传感器**: 实现 `BaseSensor` 接口 → `SensorManager.register()` → 即可参与AI决策
- **新增设备**: 实现 `BaseDevice` 接口 → `DeviceManager.register()` → ToolRegistry注册工具
- **新增Agent领域**: 添加 `agents/xxx_agent.py` → 路由规则追加关键词

### 7.3 商业价值

- **目标市场**: 全球年出货超1亿台家庭终端（中兴数据），端侧AI家居市场
- **成本优势**: 纯端侧+开源模型，零云服务费
- **隐私合规**: 数据100%本地处理，符合GDPR/个人信息保护法
- **离线可用**: 山区/地下室/断网环境正常工作
